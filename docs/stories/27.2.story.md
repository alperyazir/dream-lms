# Story 27.2: Performance Audit

## Status

Draft

## Story

**As a** platform administrator,
**I want** a performance audit identifying bottlenecks,
**so that** the system handles 5,000+ concurrent users without degradation.

## Acceptance Criteria

1. All database queries reviewed for N+1 problems and missing indexes
2. PgBouncer pool sizing validated for target concurrency (5k users)
3. Redis caching strategy documented with TTL rationale
4. Top 10 slowest endpoints identified with optimization recommendations
5. Frontend core bundle under 500KB gzipped
6. No synchronous blocking calls in async endpoints

## Tasks / Subtasks

- [ ] Task 1: Database Query Audit (AC: 1)
  - [ ] Review all SQLModel queries in `backend/app/services/` and `backend/app/api/routes/`
  - [ ] Identify N+1 query patterns (relationship access in loops without eager loading)
  - [ ] Check for missing indexes on frequently queried columns (foreign keys, search fields)
  - [ ] Review pagination — ensure large result sets aren't loaded fully into memory
  - [ ] Check for `select *` patterns that load unnecessary columns
  - [ ] Verify Alembic migrations include all necessary indexes

- [ ] Task 2: Connection Pool Audit (AC: 2)
  - [ ] Review PgBouncer config in `pgbouncer/pgbouncer.ini`
  - [ ] Validate `default_pool_size`, `max_client_conn`, `min_pool_size` for 5k users
  - [ ] Review SQLAlchemy pool settings in `backend/app/core/db.py`
  - [ ] Check for connection leaks — are sessions properly closed in all paths?
  - [ ] Verify `pool_mode = transaction` is appropriate for the query patterns used
  - [ ] Review `postgresql-tuning.conf` settings against VPS specs

- [ ] Task 3: Redis Caching Audit (AC: 3)
  - [ ] Map all cache keys and their TTLs in `backend/app/services/redis_cache.py`
  - [ ] Identify hot paths that should be cached but aren't
  - [ ] Check cache invalidation — are stale values possible after writes?
  - [ ] Review DCS cache in `backend/app/services/dcs_cache.py`
  - [ ] Document the caching strategy with rationale for each TTL
  - [ ] Check for thundering herd protection (single-flight pattern)

- [ ] Task 4: API Response Time Profiling (AC: 4)
  - [ ] Review slow request logging in `backend/app/main.py` (SLOW request middleware)
  - [ ] Identify the slowest endpoints by design (AI generation, book content loading)
  - [ ] Check DCS client timeouts and retry configuration
  - [ ] Review LLM/TTS call patterns — timeouts, fallbacks, circuit breakers
  - [ ] List top 10 endpoints likely to be slow with recommendations

- [ ] Task 5: Frontend Performance Audit (AC: 5)
  - [ ] Run `npx vite-bundle-visualizer` or check build output sizes
  - [ ] Verify route-based code splitting (lazy loading)
  - [ ] Check for large dependencies that could be tree-shaken or lazy-loaded
  - [ ] Review list rendering — are long lists virtualized?
  - [ ] Check for unnecessary re-renders (missing React.memo, useCallback)
  - [ ] Verify images are optimized and lazy-loaded

- [ ] Task 6: Async Code Audit (AC: 6)
  - [ ] Search for synchronous I/O calls in async endpoints
  - [ ] Check for blocking operations: `time.sleep`, synchronous file I/O, sync DB calls
  - [ ] Review `sync_engine` usage in `ai_generation.py` (known sync session creation)
  - [ ] Verify all external HTTP calls use async httpx (not requests)

- [ ] Task 7: Compile Performance Report (AC: 1-6)
  - [ ] Categorize findings by impact (Critical/High/Medium/Low)
  - [ ] Include specific optimization recommendations
  - [ ] Save report to `docs/qa/performance-audit-report.md`

## Dev Notes

### Key Files to Review

- **Database:** `backend/app/core/db.py`, `backend/app/models.py`, all service files
- **PgBouncer:** `pgbouncer/pgbouncer.ini`, `postgresql-tuning.conf`
- **Redis:** `backend/app/services/redis_cache.py`, `backend/app/services/dcs_cache.py`
- **Slow logging:** `backend/app/main.py` (SlowRequestMiddleware)
- **DCS client:** `backend/app/services/dream_storage_client.py` (timeouts, retries)
- **LLM manager:** `backend/app/services/llm/manager.py` (provider fallback)
- **Frontend build:** `frontend/vite.config.ts`, `frontend/package.json`

### Known Performance Concerns

- `ai_generation.py` creates a sync SQLAlchemy engine per request for skill dispatch
- DCS storage endpoints fall back to JWT auth (extra login roundtrip)
- In-memory quiz storage has no size limit
- Some services load full book content from DCS on every request

### Testing

- This is an audit story — output is a findings report, not code changes
- Report location: `docs/qa/performance-audit-report.md`
