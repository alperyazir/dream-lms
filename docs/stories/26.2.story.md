# Story 26.2: Platform Usage & Assignment Analytics Endpoints

## Status: Done

## Story

**As an** admin,
**I want** API endpoints that return platform-wide analytics (active users, assignment metrics, and AI generation stats),
**so that** I can understand how the platform is being used and make data-driven decisions.

## Acceptance Criteria

1. `GET /api/v1/admin/analytics/platform-usage` endpoint returns:
   - Daily active users (DAU), weekly active users (WAU), monthly active users (MAU) — split by students and teachers
   - Login trend data points over the selected period (daily granularity)
   - New user registration counts over the selected period
2. `GET /api/v1/admin/analytics/assignment-metrics` endpoint returns:
   - Total assignments created (with trend data over the period)
   - Platform-wide completion rate (percentage)
   - Average score across all completed assignments
   - Completion rate breakdown by activity type
   - Top 3 and bottom 3 performing activity types by average score
3. `GET /api/v1/admin/analytics/ai-usage` endpoint returns:
   - Total AI-generated activities count (with trend data over the period)
   - Breakdown by activity type (count per type)
   - Most frequently generated activity type
4. All three endpoints require admin role authentication (`AdminOrSupervisor` dependency)
5. All endpoints support a `period` query parameter: `7d`, `30d`, `90d` (default: `30d`)
6. All endpoints return responses within 500ms, using Redis caching with 5-minute TTL
7. All endpoints are rate-limited using the existing slowapi setup

## Dev Notes

**Existing Analytics Patterns (from `backend/app/services/analytics_service.py`):**
- Period filtering uses `get_period_start_date(period)` which maps `"7d"` → 7 days, `"30d"` → 30 days, `"3m"` → 90 days [Source: analytics_service.py:73-94]
- Aggregate queries use SQLModel's `func.avg()`, `func.count()`, `func.sum()` [Source: docs/architecture/7-analytics-engine.md#7.1]
- `AssignmentStudent` model has: `score`, `status` (enum: pending/in_progress/completed), `completed_at`, `student_id`, `assignment_id` [Source: backend/app/models.py]
- `AssignmentStatus` enum: `pending`, `in_progress`, `completed` [Source: backend/app/models.py]

**Redis Cache Patterns (from `backend/app/services/redis_cache.py`):**
- `cache_get_or_fetch(key, fetch_fn, ttl)` — single-flight locking, graceful degradation [Source: redis_cache.py:146+]
- `cache_get(key)` / `cache_set(key, value, ttl)` — simple get/set with jitter [Source: redis_cache.py:90-115]
- Cache keys use colon-separated namespaces (e.g., `student:{id}:analytics`)
- TTL includes ±20% jitter automatically via `cache_set()`

**Admin Route Patterns (from `backend/app/api/routes/admin.py`):**
- Router prefix: `/admin`, tags: `["admin"]` [Source: admin.py:89]
- Auth dependency: `current_user: User = AdminOrSupervisor` [Source: admin.py]
- Rate limiting: `@limiter.limit(RateLimits.ADMIN)` with `request: Request` as first param [Source: admin.py]
- Both sync (`SessionDep`) and async (`AsyncSessionDep`) sessions used — prefer `AsyncSessionDep` for new endpoints

**Key Models (from `backend/app/models.py`):**
- `User`: `id`, `email`, `username`, `role` (UserRole enum), `is_active`, `created_at`, `last_login` (check if exists)
- `Assignment`: `id`, `teacher_id`, `activity_id`, `created_at`, `due_date`
- `AssignmentStudent`: `id`, `assignment_id`, `student_id`, `score`, `status`, `completed_at`, `started_at`
- `Activity`: `id`, `activity_type`, `book_id`, `created_at` — check for AI-generation flag
- `UserRole`: admin, supervisor, publisher, teacher, student [Source: models.py:32-38]
