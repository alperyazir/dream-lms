# Story 27.3: Bug & Reliability Audit

## Status

Draft

## Story

**As a** platform administrator,
**I want** a reliability audit that finds edge cases and error handling gaps,
**so that** the system doesn't crash or corrupt data when real users hit unexpected scenarios.

## Acceptance Criteria

1. All API error responses return a consistent format (`{"detail": "..."}`)
2. No unhandled exceptions that crash the server process
3. Data integrity constraints verified (FK, unique, not null) across all models
4. Frontend handles all empty/error/loading states gracefully
5. Race conditions in concurrent operations identified and addressed
6. Background task (Arq worker) failures are logged and recoverable

## Tasks / Subtasks

- [ ] Task 1: Error Handling Audit (AC: 1, 2)
  - [ ] Review all try/except blocks — are exceptions caught too broadly (`except Exception`)?
  - [ ] Verify all API endpoints return consistent error format
  - [ ] Check for bare `except:` or `except Exception` that swallow important errors
  - [ ] Verify errors are logged with sufficient context for debugging
  - [ ] Check for missing error handling on external service calls (DCS, LLM, TTS)

- [ ] Task 2: Data Integrity Audit (AC: 3)
  - [ ] Review all SQLModel models for proper constraints (nullable, unique, FK)
  - [ ] Check cascade delete behavior — are orphan records possible?
  - [ ] Verify UUID generation is consistent (uuid4 everywhere)
  - [ ] Check for missing NOT NULL constraints on required fields
  - [ ] Review Alembic migration history for missed constraints

- [ ] Task 3: Race Condition Audit (AC: 5)
  - [ ] Review AI quota decrement — can concurrent requests overspend the quota?
  - [ ] Check assignment submission — can a student submit twice simultaneously?
  - [ ] Review Redis cache invalidation — can stale reads occur during writes?
  - [ ] Check DCS client token refresh — is the lock properly preventing concurrent refreshes?
  - [ ] Review any read-modify-write patterns without proper locking

- [ ] Task 4: Frontend State Audit (AC: 4)
  - [ ] Check all pages for empty state handling (no data)
  - [ ] Verify error states show user-friendly messages (not raw error objects)
  - [ ] Check loading states — are spinners shown during API calls?
  - [ ] Review form validation — are errors displayed before submission?
  - [ ] Check for stale data after mutations (query invalidation)
  - [ ] Verify auth redirect — what happens when token expires mid-session?

- [ ] Task 5: API Contract Audit (AC: 1, 4)
  - [ ] Compare frontend TypeScript types with backend Pydantic response models
  - [ ] Check for optional fields the frontend accesses without null checks
  - [ ] Verify pagination responses match frontend expectations
  - [ ] Check date/time format consistency (ISO 8601 everywhere?)

- [ ] Task 6: Background Task Audit (AC: 6)
  - [ ] Review Arq worker configuration and task definitions
  - [ ] Check task retry behavior on failure
  - [ ] Verify failed tasks are logged with error details
  - [ ] Check for tasks that could leave inconsistent state if interrupted
  - [ ] Review task timeout settings

- [ ] Task 7: Edge Case Audit (AC: 1-6)
  - [ ] Unicode handling in book names, teacher names, file paths
  - [ ] Large file uploads — size limits, timeout handling
  - [ ] Empty arrays/objects — do services handle zero-length inputs?
  - [ ] Concurrent user sessions — does logout invalidate all sessions?
  - [ ] Clock skew — JWT expiry with server time differences

- [ ] Task 8: Compile Reliability Report (AC: 1-6)
  - [ ] Categorize findings by severity
  - [ ] Include reproduction steps where possible
  - [ ] Save report to `docs/qa/reliability-audit-report.md`

## Dev Notes

### Key Files to Review

- **Error handling:** All route files in `backend/app/api/routes/`
- **Models/constraints:** `backend/app/models.py`
- **Background tasks:** `backend/app/tasks/`, Arq worker config
- **Redis state:** `backend/app/services/redis_cache.py`, auth cache in `deps.py`
- **Frontend types:** `frontend/src/types/`, `frontend/src/client/`
- **Frontend states:** All route components in `frontend/src/routes/`

### Known Reliability Concerns

- Cached user from Redis has no ORM relationships — any code accessing `user.teacher`, `user.publisher` etc. fails silently
- In-memory quiz storage (`InMemoryQuizStorage`) is lost on server restart
- AI quota uses read-modify-write without row-level locking
- Some DCS error responses may not match the expected format

### Testing

- This is an audit story — output is a findings report, not code changes
- Report location: `docs/qa/reliability-audit-report.md`
