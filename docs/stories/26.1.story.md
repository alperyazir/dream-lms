# Story 26.1: Uptime Kuma Integration via Docker Compose

## Status: Done

## Story

**As an** admin,
**I want** a self-hosted uptime monitoring dashboard that watches all critical services,
**so that** I can see at a glance whether the API, frontend, database, Redis, and external services are healthy.

## Acceptance Criteria

1. Uptime Kuma service is added to `docker-compose.yml` with a persistent volume for data retention across restarts
2. Uptime Kuma is accessible on port 3001 from the host
3. A `/api/v1/health` backend health endpoint exists that checks database and Redis connectivity, returning structured JSON with individual service statuses
4. The health endpoint is publicly accessible (no auth required) for monitoring tools
5. Uptime Kuma monitors are documented for setup:
   - API health endpoint (HTTP check)
   - Frontend URL (HTTP check)
   - PostgreSQL (TCP check on port 5432 via `db` hostname)
   - Redis (TCP check on port 6380 via `redis` hostname)
   - External DCS service endpoints (HTTP check)
6. Service restarts cleanly and retains all monitoring data via the persistent volume
7. No existing services are affected by the addition

## Dev Notes

**Existing Health Check:**
The backend already has a health check at `/api/v1/utils/health-check/` (referenced in `docker-compose.yml` line 167). The new `/api/v1/health` endpoint is a more detailed version for external monitoring that also checks Redis. Keep the existing one for Docker's internal healthcheck.

**Docker Compose Context:**
- Current `docker-compose.yml` structure: `db` (postgres:17 on default port), `pgbouncer` (port 6432), `redis` (redis:7-alpine on port 6380), `adminer`, `prestart`, `backend`, `arq-worker`, `frontend`
- Networks: `traefik-public` (external) and `default`
- Redis runs on **port 6380** (not default 6379) — Uptime Kuma TCP check must target port 6380
- Backend connects to PostgreSQL through PgBouncer on port 6432, but Uptime Kuma should check the actual `db` service on port 5432 directly
- Existing volumes: `app-db-data`, `redis-data`

**Redis Cache Pattern (from `backend/app/services/redis_cache.py`):**
- Get Redis client via `get_redis()` which returns `redis.Redis | None`
- Graceful degradation: if Redis is down, returns None — use this pattern in health check
- `await redis_client.ping()` to verify connectivity

**Backend API Router Registration (from `backend/app/api/main.py`):**
- Routes are registered on the main API router: `api_router.include_router(health.router)`
- All routes are under `/api/v1/` prefix

**Rate Limiting (from `backend/app/core/rate_limit.py`):**
- Uses `slowapi` with Redis backend
- Pattern: `@limiter.limit(RateLimits.SOME_LIMIT)` decorator on route functions
- Request object must be first parameter for limiter to work

**Coding Standards:**
- Python: snake_case, type hints, async/await, Google-style docstrings [Source: docs/architecture/coding-standards.md]
- FastAPI route pattern: typed responses, proper status codes [Source: docs/architecture/coding-standards.md#FastAPI Route Patterns]
- Error handling: use HTTPException with proper status codes [Source: docs/architecture/coding-standards.md#Error Handling]
