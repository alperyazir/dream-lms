# Story 27.4: Fix Critical & High Findings

## Status

Draft

## Story

**As a** development team,
**I want** to fix all critical and high-severity findings from the security, performance, and reliability audits,
**so that** the system is safe and stable for production launch.

## Acceptance Criteria

1. Zero Critical severity findings remaining from audit reports
2. Zero High severity findings remaining from audit reports
3. Medium/Low findings documented in backlog (GitHub issues or docs/qa/tech-debt.md)
4. All fixes verified by re-running the relevant audit check
5. No regressions introduced — existing functionality still works

## Tasks / Subtasks

- [ ] Task 1: Triage audit findings (AC: 1, 2, 3)
  - [ ] Read `docs/qa/security-audit-report.md`
  - [ ] Read `docs/qa/performance-audit-report.md`
  - [ ] Read `docs/qa/reliability-audit-report.md`
  - [ ] Create prioritized fix list: Critical first, then High
  - [ ] Document Medium/Low as tech debt in `docs/qa/tech-debt.md`

- [ ] Task 2: Fix Critical findings (AC: 1)
  - [ ] Fix each Critical finding per audit reports
  - [ ] Verify fix resolves the issue
  - [ ] Note: Specific subtasks depend on audit report output

- [ ] Task 3: Fix High findings (AC: 2)
  - [ ] Fix each High finding per audit reports
  - [ ] Verify fix resolves the issue
  - [ ] Note: Specific subtasks depend on audit report output

- [ ] Task 4: Regression verification (AC: 5)
  - [ ] Run backend tests: `pytest`
  - [ ] Run frontend build: `npm run build`
  - [ ] Run linters: `ruff check .` and `eslint src/`
  - [ ] Manually verify core flows still work (login, books, assignments)

- [ ] Task 5: Re-run audit checks (AC: 4)
  - [ ] Re-verify each fixed finding is resolved
  - [ ] Update audit reports with fix status
