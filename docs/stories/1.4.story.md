# Story 1.4: Admin User Management Interface

## Status
**Done**

## Story

**As an** admin,
**I want** to create, view, edit, and delete all user types (publishers, teachers, students) through a web interface,
**so that** I can manage the entire system and onboard new organizations.

## Acceptance Criteria

1. Admin dashboard displays system statistics: total publishers, schools, teachers, students
2. Admin can navigate to Publishers list view showing all publishers with search and pagination
3. Admin can create new publisher via form modal (fields: name, contact_email)
4. Admin can edit existing publisher details via form modal
5. Admin can delete publisher (soft delete) with confirmation dialog
6. Admin can navigate to Schools list view showing all schools with associated publisher name
7. Admin can create new school via form modal (fields: name, publisher selection, address, contact_info)
8. Admin can edit/delete schools with appropriate confirmations
9. Admin can navigate to Teachers list view showing all teachers with school and publisher info
10. Admin can navigate to Students list view showing all students with associated teacher/school info
11. All list views include search functionality and column sorting
12. API endpoints for admin operations verify admin role via `require_role` dependency
13. Frontend displays appropriate error messages for failed operations
14. Responsive design works on desktop and tablet screen sizes
15. Unit and integration tests cover all admin CRUD operations

### Database Schema Information

[Source: architecture/2-database-schema-design.md#2.2-Complete-Table-Definitions]

**Relevant Tables for This Story:**

**`users` table (lines 193-208):**
- id (UUID, PK)
- email (VARCHAR, UNIQUE, NOT NULL)
- password_hash (VARCHAR, NOT NULL)
- role (ENUM: 'admin', 'publisher', 'teacher', 'student')
- is_active (BOOLEAN, default true) - used for soft deletes
- created_at, updated_at (TIMESTAMP WITH TIME ZONE)
- Email format constraint enforced at DB level

**`publishers` table (lines 213-223):**
- id (UUID, PK)
- user_id (UUID, UNIQUE, FK to users.id, CASCADE DELETE)
- name (VARCHAR 255, NOT NULL)
- contact_email (VARCHAR 255, nullable)
- created_at, updated_at (TIMESTAMP WITH TIME ZONE)

**`schools` table (lines 228-239):**
- id (UUID, PK)
- name (VARCHAR 255, NOT NULL)
- publisher_id (UUID, FK to publishers.id, CASCADE DELETE)
- address (TEXT, nullable)
- contact_info (TEXT, nullable)
- created_at, updated_at (TIMESTAMP WITH TIME ZONE)

**`teachers` table (lines 244-255):**
- id (UUID, PK)
- user_id (UUID, UNIQUE, FK to users.id, CASCADE DELETE)
- school_id (UUID, FK to schools.id, CASCADE DELETE)
- subject_specialization (VARCHAR 255, nullable)
- created_at, updated_at (TIMESTAMP WITH TIME ZONE)

**`students` table (lines 260-270):**
- id (UUID, PK)
- user_id (UUID, UNIQUE, FK to users.id, CASCADE DELETE)
- grade_level (VARCHAR 50, nullable)
- parent_email (VARCHAR 255, nullable)
- created_at, updated_at (TIMESTAMP WITH TIME ZONE)

**Important Relationships:**
- Publishers → Schools (one-to-many via `schools.publisher_id`)
- Schools → Teachers (one-to-many via `teachers.school_id`)
- Users → Role-specific tables (one-to-one via `user_id`)
- Soft deletes use `users.is_active` flag (don't physically delete records)

### Important Implementation Notes

1. **Soft Deletes**: Use `is_active=False` on the `users` table instead of physical deletes - this preserves referential integrity and allows for audit trails

2. **Password Generation**: When admin creates a publisher, auto-generate a temporary password and display it once (similar to what publishers/teachers will do for their sub-users in future stories)

3. **Cascading Relationships**: When deleting a publisher, associated schools/teachers/students may cascade delete - ensure proper confirmation dialogs warn about this

4. **Search Implementation**: Implement search as query parameter filtering on the backend (e.g., `?search=name`), use SQLAlchemy `ilike()` for case-insensitive matching

5. **Pagination**: Use offset/limit pagination (e.g., `?page=1&per_page=20`), return pagination metadata in response

6. **Form Validation**: Use Zod schemas on frontend that mirror Pydantic schemas on backend for consistent validation

7. **Responsive Design**: Admin interface should work on desktop (1920x1080) and tablet (768x1024) but mobile is not a priority for admin features

8. **Role Guard**: Ensure `RoleGuard` component redirects non-admin users to their appropriate dashboards, not to login page

9. **API Response Unwrapping**: The axios interceptor in `services/api.ts` unwraps `response.data.data`, so service functions should return just the data

10. **Loading States**: Show skeleton loaders for tables during data fetching, disable buttons during mutations

### Top Issues Identified

**Issue #1: Inconsistent Soft Delete Implementation**
- **Severity**: Medium
- **Location**: `backend/app/services/admin_service.py:479-501`
- **Finding**: Publishers use soft delete (setting `user.is_active=False`) which preserves audit trail, but schools use hard delete (`await db.delete(school)`) which physically removes records. This inconsistency could lead to referential integrity issues and loss of audit trail.
- **Code Comment Found**: Line 499 states "Note: If we add is_active to School model, we can soft delete instead"
- **Impact**: Loss of historical data when schools are deleted, potential cascade deletion issues for associated teachers
- **Recommended Action**: Add `is_active` or `deleted_at` field to School model and implement soft delete pattern consistently across all entities
- **Suggested Owner**: dev

**Issue #2: Missing Input Validation**
- **Severity**: Medium
- **Location**: `backend/app/schemas/admin.py` (PublisherCreate, SchoolCreate)
- **Finding**: No minimum length validation on publisher name (line 27) or school name (line 61). Empty strings could technically pass validation.
- **Impact**: Potential data quality issues, empty names could break UI assumptions
- **Recommended Action**: Add Pydantic field validators with `min_length=1` or `min_length=3` as appropriate
- **Suggested Owner**: dev

**Issue #3: Frontend Tests Deferred**
- **Severity**: Medium
- **Location**: Frontend test files (Tasks 16-17 in story)
- **Finding**: All frontend component and hook tests were deferred to a future story. This leaves the frontend with 0% test coverage.
- **Impact**: No automated validation of UI behavior, higher risk of regressions in future changes
- **Recommended Action**: Prioritize frontend test coverage in next sprint (component tests for PublishersPage, SchoolsPage, and hook tests for CRUD operations)
- **Suggested Owner**: dev

**Issue #4: Pydantic Deprecation Warnings**
- **Severity**: Low
- **Location**: Multiple schema files using `class Config` pattern
- **Finding**: 4 Pydantic deprecation warnings about class-based Config being deprecated in favor of ConfigDict
- **Impact**: Code will break in Pydantic V3.0
- **Recommended Action**: Migrate from `class Config` to `model_config = ConfigDict(from_attributes=True)`
- **Suggested Owner**: dev
